Version 11.9.4 November 7, 2008
-------------------------------

    1. Problem:
               Driver takes up to two minutes or longer to load when
               loading from floppy drive.
      Cause:
               Delay macro MM_Wait() uses 16 bit value instead of 32 bit 
               value as the actual delay routine MM_Stall() expected. When
               booting up from a floppy drive, the upper 16 bit value on the
               stack happen to be non zero, so the wait takes much longer than
               anticipated.
      Change:
               Removed the use of the macro. Use the actual delay routine.
      Impact:
               None.

Version 10.4.5 May 23, 2007
---------------------------

   1. Problem:
               CQ #28789: DOS NDIS2 Ghostcast session (with a MAC/DIS to Packet
               Driver Converter) is running at 20%-25% slower than a third party NIC.

      Cause:
               The UDP stream causes the NDIS2 driver to drop packets due to lack
               of receive buffer (one Receive buffer) and driver is Config Cycle
               dependency.

      Change: 
               Add one additional receive buffer and add keyword NoTagStatus. The
               NoTagStatus will sets the driver NOT to utilize the default method
               Tag Status of handling interrupts. This will reduce the number of
               Config Cycle Read/Write in the ISR Loop. Hence, speed up the packet
               receiving process.

      Impact:
               Driver size is increased by 2KB.

   Enhancement:
   ============

   1. Request:
               Remove support for the following device:
                BCM5756(BCM5755ME).

     Change:
               Removed support for the following device:
                BCM5756(BCM5755ME).

     Impact:
               None.

   Note:
   =====

   1. Use this driver in conjunction with the keyword NoTagStatus in the Protocol.ini
      file will increase the Ghostcast session (with a MAC/DIS to Packet Driver
      Converter) performance by 10%-15%.

Version 9.13 October 02, 2006
-----------------------------

   1. Problem:
               CQ #26644: NDIS2 driver causes OEM platform Point Of Sale (POS)
               terminals to flood the network with Remote Program Load (RPL)
               packets.

      Cause:
               The fix in NDIS2 v8.15 that resolved CQ #12373 which will forward
               all the Multicast traffic to the upper protocol in used: TCP/IP.
               This causes the upper protocol responses to a Logical Link Control
               (LLC) Multicast packet with a XID packet. This XID response packet
               triggers the Point Of Sale terminals to flood the network with
               RPL packets.

      Change:
               The NDIS2 driver is NOT to deliver Multicast packets to the upper
               protocol when the Multicast Address Table is empty (as before v8.15).
               Also, add a Keyword "AcceptAllMC" in the PROTOCOL.INI file to inform
               the NDIS2 driver to deliver ALL Multicast packets to the upper
